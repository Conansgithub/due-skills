# 节点服务器 (Node)

## 概述

节点服务器（node）是 Due 框架集群架构的核心业务处理单元，负责处理客户端请求、执行业务逻辑、管理用户状态等。采用模块化设计，支持多种调度模型。

## 包路径

```go
import "github.com/dobyte/due/v2/cluster/node"
```

## 配置参数

```toml
[cluster.node]
    # 实例ID，集群中唯一。不填写默认自动生成唯一的实例ID
    id = ""
    # 实例名称
    name = "node"
    # 内建RPC服务器监听地址。不填写默认随机监听
    addr = ":0"
    # 是否将内部通信地址暴露到公网。默认为false
    expose = false
    # 编解码器。可选：json | proto。默认为proto
    codec = "json"
    # RPC调用超时时间，支持单位：纳秒（ns）、微秒（us | µs）、毫秒（ms）、秒（s）、分（m）、小时（h）、天（d）。默认为3s
    timeout = "3s"
    # 节点权重，用于节点无状态路由消息的加权轮询策略，权重值必需大于0才生效。默认为1
    weight = 1
    # 实例元数据
    [cluster.node.metadata]
        # 键值对，且均为字符串类型
        key = "value"

[locate.redis]
    # 客户端连接地址
    addrs = ["127.0.0.1:6379"]
    # 数据库号
    db = 0
    # 用户名
    username = ""
    # 密码
    password = ""
    # 最大重试次数
    maxRetries = 3
    # key前缀
    prefix = "due:locate"

[packet]
    # 字节序，默认为big。可选：little | big
    byteOrder = "big"
    # 路由字节数，默认为2字节
    routeBytes = 2
    # 序列号字节数，默认为2字节
    seqBytes = 2
    # 消息字节数，默认为5000字节
    bufferBytes = 5000
```

## 调度模型

节点服务器支持三种调度模型：

1. **单线程模型**：在路由处理器中直接处理消息
2. **多协程模型**：使用协程池异步处理消息
3. **Actor模型**：使用Actor模式处理消息

## 使用示例

### 基础节点服务器

```go
package main

import (
    "fmt"
    "github.com/dobyte/due/locate/redis/v2"
    "github.com/dobyte/due/registry/consul/v2"
    "github.com/dobyte/due/v2"
    "github.com/dobyte/due/v2/cluster/node"
    "github.com/dobyte/due/v2/codes"
    "github.com/dobyte/due/v2/log"
    "github.com/dobyte/due/v2/utils/xtime"
)

// 路由号
const greet = 1

func main() {
    // 创建容器
    container := due.NewContainer()
    // 创建用户定位器
    locator := redis.NewLocator()
    // 创建服务发现
    registry := consul.NewRegistry()
    // 创建节点组件
    component := node.NewNode(
        node.WithLocator(locator),
        node.WithRegistry(registry),
    )
    // 初始化应用
    initApp(component.Proxy())
    // 添加节点组件
    container.Add(component)
    // 启动容器
    container.Serve()
}

// 初始化应用
func initApp(proxy *node.Proxy) {
    proxy.Router().AddRouteHandler(greet, greetHandler)
}

// 请求
type greetReq struct {
    Message string `json:"message"`
}

// 响应
type greetRes struct {
    Code    int    `json:"code"`
    Message string `json:"message"`
}

// 路由处理器
func greetHandler(ctx node.Context) {
    req := &greetReq{}
    res := &greetRes{}
    ctx.Defer(func() {
        if err := ctx.Response(res); err != nil {
            log.Errorf("response message failed: %v", err)
        }
    })

    if err := ctx.Parse(req); err != nil {
        log.Errorf("parse request message failed: %v", err)
        res.Code = codes.InternalError.Code()
        return
    }

    log.Info(req.Message)

    res.Code = codes.OK.Code()
    res.Message = fmt.Sprintf("I'm tcp server, and the current time is: %s", xtime.Now().Format(xtime.DateTime))
}
```

### 多协程模型

```go
func greetHandler(ctx node.Context) {
    // 投递到协程池异步处理
    ctx.Task(func() {
        req := &greetReq{}
        res := &greetRes{}
        
        if err := ctx.Parse(req); err != nil {
            log.Errorf("parse request failed: %v", err)
            res.Code = codes.InternalError.Code()
            ctx.Response(res)
            return
        }
        
        // 执行业务逻辑
        result := doBusinessLogic(req)
        
        res.Code = codes.OK.Code()
        res.Message = result
        ctx.Response(res)
    })
}
```

### Actor模型

```go
func greetHandler(ctx node.Context) {
    // 投递到Actor处理
    ctx.Actor(func() {
        req := &greetReq{}
        res := &greetRes{}
        
        if err := ctx.Parse(req); err != nil {
            log.Errorf("parse request failed: %v", err)
            res.Code = codes.InternalError.Code()
            ctx.Response(res)
            return
        }
        
        // Actor内部处理逻辑
        result := processInActor(req)
        
        res.Code = codes.OK.Code()
        res.Message = result
        ctx.Response(res)
    })
}
```

## 最佳实践

1. **调度模型选择**:
   - 简单业务使用单线程模型
   - IO密集型业务使用多协程模型
   - 需要状态隔离的业务使用Actor模型

2. **错误处理**:
   - 使用 `ctx.Defer` 确保响应一定发送
   - 统一错误码规范

3. **性能优化**:
   - 合理设置节点权重实现负载均衡
   - 使用连接池减少连接开销

4. **状态管理**:
   - 使用用户定位器追踪用户位置
   - 合理设计消息路由规则
