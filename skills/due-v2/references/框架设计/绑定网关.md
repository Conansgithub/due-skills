# 绑定网关

## 概述

绑定网关（Bind Gate）是用户登录流程中的关键步骤，用于建立用户（UID）与网关连接（Gate Connection）的关联关系。只有完成绑定，网关才能将用户的消息正确路由到节点服，并将节点服的响应转发给用户。

## 核心概念

### 三元组关系

```
┌─────────┐         ┌─────────┐         ┌─────────┐
│   UID   │ ←────── │  Gate   │ ←────── │  Conn   │
│ 用户 ID │  绑定    │ 网关服  │  连接    │ 连接 ID │
└─────────┘         └─────────┘         └─────────┘
     ↑                                     ↑
     │                                     │
   业务层标识                           网络层标识
```

- **UID（User ID）**：用户唯一标识，由业务层生成
- **Gate**：网关服实例，维护连接池
- **Conn（Connection ID）**：连接标识，由网关服分配

### 绑定信息

绑定信息存储在 Redis 中，包含以下内容：

```go
type GateBinding struct {
    UID    int64  `json:"uid"`    // 用户 ID
    GateID string `json:"gateID"` // 网关服 ID
    ConnID int64  `json:"connID"` // 连接 ID
}
```

Redis Key 格式：
```
due:locate:user:{uid}
```

## 绑定流程

### 完整登录流程

```
1. 客户端建立 TCP/WebSocket 连接
   ↓
2. 网关服分配 Connection ID（临时连接，未绑定用户）
   ↓
3. 客户端发送登录请求（route=1, 携带账号密码）
   ↓
4. 网关服将消息转发到节点服（无状态路由）
   ↓
5. 节点服验证账号密码
   ↓
6. 验证成功，生成 UID
   ↓
7. 节点服调用 ctx.BindGate(uid) 绑定网关
   ↓
8. 绑定信息写入 Redis
   ↓
9. 节点服返回登录成功响应
   ↓
10. 网关服根据绑定信息将响应转发给客户端
```

### 代码示例

```go
// 登录处理器（无状态路由）
func loginHandler(ctx node.Context) {
    req := &loginReq{}
    if err := ctx.Parse(req); err != nil {
        ctx.Response(&loginRes{Code: 400, Message: "参数错误"})
        return
    }
    
    // 1. 验证账号密码
    uid, err := verifyUser(req.Account, req.Password)
    if err != nil {
        ctx.Response(&loginRes{Code: 401, Message: "账号或密码错误"})
        return
    }
    
    // 2. 绑定网关（关键步骤）
    if err := ctx.BindGate(uid); err != nil {
        log.Errorf("绑定网关失败: uid=%d, err=%v", uid, err)
        ctx.Response(&loginRes{Code: 500, Message: "服务器内部错误"})
        return
    }
    
    // 3. 绑定节点（如果是有状态服务）
    if err := ctx.BindNode(uid); err != nil {
        log.Errorf("绑定节点失败: uid=%d, err=%v", uid, err)
        ctx.UnbindGate(uid)  // 回滚网关绑定
        ctx.Response(&loginRes{Code: 500, Message: "服务器内部错误"})
        return
    }
    
    // 4. 初始化玩家数据
    if err := initPlayerData(uid); err != nil {
        log.Errorf("初始化玩家数据失败: uid=%d, err=%v", uid, err)
        ctx.UnbindNode(uid)
        ctx.UnbindGate(uid)
        ctx.Response(&loginRes{Code: 500, Message: "服务器内部错误"})
        return
    }
    
    // 5. 返回登录成功
    ctx.Response(&loginRes{
        Code:    0,
        Message: "登录成功",
        UID:     uid,
    })
}
```

## 绑定机制详解

### BindGate 方法

```go
func (ctx *Context) BindGate(uid int64) error
```

**参数：**
- `uid`：用户 ID，必须是全局唯一的正整数

**返回值：**
- `error`：绑定失败时返回错误

**内部实现：**

```go
func (ctx *Context) BindGate(uid int64) error {
    // 1. 检查 UID 合法性
    if uid <= 0 {
        return errors.New("UID 必须大于 0")
    }
    
    // 2. 检查是否已绑定
    if ctx.UID() != 0 {
        return errors.New("当前连接已绑定用户")
    }
    
    // 3. 检查 UID 是否已在其他连接登录
    existing, err := ctx.locator.Locate(uid)
    if err == nil && existing != nil {
        // 踢下线：断开旧连接
        ctx.gate.Kick(existing.GateID, existing.ConnID)
    }
    
    // 4. 写入绑定信息到 Redis
    binding := &GateBinding{
        UID:    uid,
        GateID: ctx.gate.ID(),
        ConnID: ctx.conn.ID(),
    }
    if err := ctx.locator.Bind(uid, binding); err != nil {
        return err
    }
    
    // 5. 更新上下文
    ctx.uid = uid
    
    return nil
}
```

### 踢下线机制

当同一账号在多个设备登录时，新登录会踢下线旧连接：

```
用户 A 在设备 1 登录（UID=1001, Gate-1, Conn-100）
    ↓
用户 A 在设备 2 登录（UID=1001, Gate-2, Conn-200）
    ↓
节点服调用 BindGate(1001)
    ↓
定位器发现 UID=1001 已绑定到 Gate-1/Conn-100
    ↓
节点服向 Gate-1 发送踢下线请求
    ↓
Gate-1 断开 Conn-100 连接
    ↓
Gate-1 向设备 1 发送"账号在其他设备登录"通知
    ↓
更新绑定信息：UID=1001 → Gate-2/Conn-200
```

**踢下线代码：**

```go
// 网关服处理踢下线请求
func (g *Gate) handleKick(req *KickReq) {
    conn := g.getConnection(req.ConnID)
    if conn == nil {
        return
    }
    
    // 发送踢下线通知
    conn.Send(&KickNotify{
        Reason: "账号在其他设备登录",
    })
    
    // 断开连接
    conn.Close()
}
```

## 解绑网关

### 自动解绑

连接断开时，网关服自动解绑：

```go
// 网关服连接关闭回调
func (g *Gate) onConnectionClose(conn *Connection) {
    uid := conn.UID()
    if uid != 0 {
        // 从定位器删除绑定信息
        g.locator.Unbind(uid)
        
        // 通知节点服用户下线
        g.notifyUserOffline(uid)
    }
}
```

### 手动解绑

节点服可以主动解绑：

```go
// 登出处理器
func logoutHandler(ctx node.Context) {
    uid := ctx.UID()
    
    // 保存玩家数据
    savePlayerData(uid)
    
    // 解绑节点
    ctx.UnbindNode(uid)
    
    // 解绑网关
    ctx.UnbindGate(uid)
    
    ctx.Response(&logoutRes{Code: 0})
}
```

```go
        log.Errorf("查询用户位置失败: uid=%d, err=%v", uid, err)

```

### 判断用户是否在线

```go
func isUserOnline(ctx node.Context, uid int64) bool {
    binding, err := ctx.Locate(uid)
    if err != nil {
        return false
    }
    return binding != nil
}
```

## 消息转发机制

### 客户端 → 节点服

```
客户端发送消息（route=10, data={...}）
    ↓
网关服接收消息
    ↓
网关服查询连接绑定信息：Conn-100 → UID=1001
    ↓
网关服构造内部消息：{UID: 1001, Route: 10, Data: {...}}
    ↓
网关服根据路由号查询节点服（无状态/有状态路由）
    ↓
网关服通过 RPC 转发消息到节点服
    ↓
节点服处理消息
```

### 节点服 → 客户端

```
节点服发送消息（UID=1001, route=10, data={...}）
    ↓
节点服查询用户绑定信息：UID=1001 → Gate-1/Conn-100
    ↓
节点服通过 RPC 发送消息到 Gate-1
    ↓
Gate-1 查找连接：Conn-100
    ↓
Gate-1 将消息转发给客户端
```

## 最佳实践

### 1. 绑定顺序

```go
// ✅ 正确顺序：先绑定网关，再绑定节点
ctx.BindGate(uid)
ctx.BindNode(uid)

// ❌ 错误顺序：先绑定节点，再绑定网关
ctx.BindNode(uid)  // 此时 UID 还未绑定网关，无法接收消息
ctx.BindGate(uid)
```

### 2. 错误处理与回滚

```go
func loginHandler(ctx node.Context) {
    uid := generateUID()
    
    // 绑定网关
    if err := ctx.BindGate(uid); err != nil {
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    // 绑定节点
    if err := ctx.BindNode(uid); err != nil {
        ctx.UnbindGate(uid)  // 回滚网关绑定
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    // 初始化数据
    if err := initPlayerData(uid); err != nil {
        ctx.UnbindNode(uid)  // 回滚节点绑定
        ctx.UnbindGate(uid)  // 回滚网关绑定
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    ctx.Response(&loginRes{Code: 0})
}
```

## 常见问题

### Q1: 绑定网关失败怎么办？

**A:** 常见原因：
- UID 不合法（<= 0）
- 当前连接已绑定其他用户
- Redis 连接失败

检查日志和 Redis 状态。

### Q2: 用户登录后收不到消息？

**A:** 可能原因：
- 未绑定网关
- 绑定信息未同步到 Redis
- 网关服与节点服 RPC 连接失败

使用 `ctx.Locate(uid)` 检查绑定信息。

### Q3: 如何实现多设备同时在线？

**A:** 默认情况下，同一 UID 只能在一个设备登录。如需支持多设备：
- 修改定位器，允许同一 UID 绑定多个连接
- 修改踢下线逻辑，不踢旧连接

### Q4: 绑定信息丢失怎么办？

**A:** 绑定信息存储在 Redis 中，如果 Redis 数据丢失：
- 用户需要重新登录
- 建议在 Redis 中启用持久化（RDB/AOF）

## 总结

绑定网关是用户登录流程的核心环节，通过建立 UID 与网关连接的关联，实现了消息的正确路由。理解绑定机制，是开发稳定可靠的游戏服务器的基础。
