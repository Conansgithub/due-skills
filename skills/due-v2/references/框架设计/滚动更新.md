# 滚动更新

## 概述

在游戏服务器开发中，完美的热更新非常困难。Due v2 框架提供了一种相对完善的解决方案，可以在不影响游戏运行的情况下对服务器进行滚动更新。框架针对网关服（Gate）、节点服（Node）和网格服（Mesh）分别设计了不同的更新策略。

## 网关服（Gate）滚动更新

### 更新原理

网关服仅做路由消息转发，不提供业务逻辑开发接口。因此网关服的更新通常不是必须的，但框架仍提供了完善的滚动更新方案。

### 更新流程

```
1. 启动新 Gate → 注册到 Registry
2. 旧 Gate 执行 kill → 收到关闭信号
3. 旧 Gate 状态变为 Hang → 同步到 Registry
4. 新用户/断线重连用户 → 获取非 Hang 状态的 Gate 地址
5. 新用户连接新 Gate，旧连接不受影响
6. 旧 Gate 所有连接断开后自动销毁
```

### 详细步骤

1. **启动新网关服**：准备一台新的 Gate 并启动，框架底层自动注册新网关服务到注册中心（Registry）
2. **关闭旧网关服**：在旧 Gate 上执行 `kill` 命令，框架底层接收关闭信号并启动关闭流程
3. **状态变更**：旧 Gate 将自身状态变更为挂起（`cluster.Hang`），并同步到注册中心
4. **流量切换**：新用户或断线老用户重新走登录流程，获取到最新非挂起状态的网关服务器地址
5. **连接迁移**：新用户通过返回的连接地址与新 Gate 建立连接，旧 Gate 上的已有连接不受影响
6. **旧服销毁**：旧 Gate 在所有连接断开后或到达最大等待时间（`etc.shutdownMaxWaitTime`）后自动销毁

### 流量控制

- **登录服控制**：通过登录服返回的连接地址控制流量分配（传统方案）
- **K8S 控制**：使用 K8S 部署时，K8S 自动将流量分配到新 Gate，无需登录服参与

## 节点服（Node）滚动更新

### 更新原理

节点服多用于有状态服务的开发，需要在保证用户体验和数据安全的前提下进行更新维护。

### 更新流程

```
1. 启动新 Node → 注册到 Registry
2. 旧 Node 执行 kill → 收到关闭信号
3. 旧 Node 状态变为 Hang → 同步到 Registry
4. 旧 Node 同步阻塞，等待内部业务处理完成
5. Gate 更新路由表：无状态路由忽略 Hang 节点，有状态路由仍按绑定关系到达
6. 旧 Node 不再接收新玩家，老玩家业务完成后自动销毁
```

### 详细步骤

1. **启动新节点服**：准备一台新的 Node 并启动，框架底层自动注册新节点服务到注册中心
2. **关闭旧节点服**：在旧 Node 上执行 `kill` 命令，框架接收关闭信号并启动关闭流程
3. **状态变更**：旧 Node 将自身状态变更为挂起（`cluster.Hang`），并同步到注册中心
4. **等待业务完成**：旧 Node 同步阻塞，等待**内部业务**处理完成
5. **路由更新**：Gate 接收到 Node 状态变更后更新本地路由表：
   - **无状态路由**：忽略处于挂起状态的 Node
   - **有状态路由**：依然根据绑定关系到达相应的 Node
6. **旧服销毁**：旧 Node 在完成老玩家业务后或到达最大等待时间（`etc.shutdownMaxWaitTime`）后自动销毁

### 内部业务定义

以下业务被视为**内部业务**，关闭流程会等待其完成：

| 类型 | 创建方式 | 说明 |
|------|---------|------|
| 绑定用户 | `ctx.BindNode()` / `proxy.BindNode()` | 绑定到节点服的用户 |
| Actor 实例 | `ctx.Spawn()` / `proxy.Spawn()` | 创建的 Actor 实例 |
| 协程任务 | `ctx.Task()` | 投递到任务池的任务 |
| 调用队列 | `proxy.Invoke()` | 投递到队列中的处理函数 |
| 延迟处理 | `proxy.AfterFunc()` | 延迟处理函数 |
| 延迟调用 | `proxy.AfterInvoke()` | 延迟调用处理函数 |

### 路由消息不算内部业务

路由消息不作为内部业务的原因是：Node 处于挂起状态时，仅处理绑定到本节点的用户路由消息，因此只需关注节点服绑定的用户数即可。

## 网格服（Mesh）更新

网格服被定义为无状态的微服务，更新没有任何限制，可以随时进行。

### 建议策略

利用 Mesh 无状态的特性，合理规划业务模块：
- 将无状态业务规划到 Mesh 上
- 将频繁改动的业务规划到 Mesh 上
- 为后续业务扩展和更新做好准备

## 配置项

### shutdownMaxWaitTime

控制服务器关闭时的最大等待时间：

```toml
# 容器关闭最大等待时间
# 支持单位：ns、us、ms、s、m、h、d
# 默认为 0（不等待）
shutdownMaxWaitTime = "30s"
```

当设置为非零值时，即使内部业务未完成，到达最大等待时间后也会强制关闭。

## 更新策略对比

| 组件 | 状态 | 新连接 | 已有连接 | 等待机制 |
|------|------|--------|---------|---------|
| Gate | Hang 后不再接收新连接 | 路由到新 Gate | 保持到断开 | 等待所有连接断开 |
| Node | Hang 后不接收新玩家 | 路由到新 Node | 按绑定继续 | 等待内部业务完成 |
| Mesh | 无限制 | 直接路由 | N/A | 无需等待 |

## 最佳实践

1. **先启新后关旧**：确保新实例已注册到注册中心后再关闭旧实例
2. **设置合理的等待时间**：`shutdownMaxWaitTime` 应根据业务处理时间合理设置
3. **网关服优先使用 K8S**：如使用 K8S 部署，可利用其内置的流量管理能力
4. **节点服数据持久化**：确保在更新前所有重要数据已持久化
5. **网格服独立部署**：将频繁更新的无状态业务放到 Mesh，减少对 Node 的更新需求
