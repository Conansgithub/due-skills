# 架构设计

## 概述

Due v2 采用分层架构设计，将游戏服务器拆分为多个独立组件，每个组件职责明确、松耦合，支持灵活部署和水平扩展。

## 核心组件

### 1. 网关服（Gate）

**职责：**
- 维护客户端长连接（TCP/WebSocket）
- 处理心跳检测与连接保活
- 消息路由与转发
- 协议编解码

**特点：**
- 无状态设计，可水平扩展
- 不提供业务开发接口，专注连接管理
- 支持多种网络协议（TCP、WebSocket）

### 2. 节点服（Node）

**职责：**
- 承载游戏业务逻辑
- 处理有状态路由消息
- 管理用户会话与游戏数据
- 提供 Actor 模型支持

**特点：**
- 有状态服务，需绑定用户（UID）
- 支持单线程、多协程、Actor 三种并发模型
- 通过注册中心实现服务发现

### 3. 网格服（Mesh）

**职责：**
- 承载无状态微服务
- 处理跨节点通信
- 提供通用服务接口（如匹配、聊天等）

**特点：**
- 无状态设计，可随时更新
- 支持 RPC 调用
- 适合频繁变动的业务模块

## 架构分层

```
┌─────────────────────────────────────────┐
│           客户端层 (Client)              │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│         网关层 (Gate Cluster)            │
│  - 连接管理、协议编解码、消息路由          │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│         业务层 (Node Cluster)            │
│  - 游戏逻辑、状态管理、Actor 模型         │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│         服务层 (Mesh Cluster)            │
│  - 无状态微服务、跨节点通信               │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│       基础设施层 (Infrastructure)        │
│  - 注册中心、配置中心、缓存、数据库        │
└─────────────────────────────────────────┘
```

## 核心设计理念

### 1. 高内聚低耦合

- 各组件职责单一，边界清晰
- 通过接口和协议通信，避免直接依赖
- 支持独立部署和升级

### 2. 模块化设计

- 核心功能模块化，可按需组合
- 支持插件化扩展（网络、存储、加密等）
- 统一接口规范，便于替换实现

### 3. 分布式友好

- 内置服务注册与发现机制
- 支持多种注册中心（Consul、Etcd、Nacos）
- 提供用户定位与消息路由能力

### 4. 灵活的并发模型

- 单线程模型：保证顺序执行，适合帧同步
- 多协程模型：提升并发能力，适合 IO 密集型
- Actor 模型：避免数据竞争，适合房间类游戏

## 关键特性

### 1. 服务注册与发现

- 节点启动时自动注册到注册中心
- 网关通过注册中心发现可用节点
- 支持健康检查与自动摘除

### 2. 用户定位

- 通过 Redis 等存储记录用户所在节点
- 支持跨节点消息路由
- 保证有状态消息准确送达

### 3. 滚动更新

- 支持不停服更新
- 网关服：新节点接管，旧节点优雅下线
- 节点服：等待业务处理完成后下线

### 4. 协议扩展

- 支持自定义协议格式
- 内置心跳与数据包协议
- 支持加密与压缩

## 技术栈

- **语言：** Go 1.18+
- **网络：** TCP、WebSocket
- **序列化：** JSON、Protobuf
- **注册中心：** Consul、Etcd、Nacos
- **存储：** Redis、MySQL、MongoDB
- **传输：** gRPC、RPCX

## 适用场景

- MMORPG（大型多人在线角色扮演游戏）
- MOBA（多人在线战术竞技游戏）
- 卡牌游戏
- 休闲竞技游戏
- 实时对战游戏
- 需要高并发、低延迟的实时应用

## 总结

Due v2 的架构设计充分考虑了游戏服务器的复杂性和多样性需求，通过分层架构、模块化设计、灵活的并发模型，为开发者提供了一个强大而灵活的开发框架。无论是小型独立游戏还是大型商业项目，都能找到合适的解决方案。
