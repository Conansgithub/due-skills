# 单线程模型

## 概述

Due v2 框架支持单线程并发模型，适用于需要严格保证执行顺序、高并发低延迟、非阻塞 IO 密集型等场景。在单线程模型下，所有路由处理器在同一线程中顺序执行，不存在并发问题，无需使用同步原语。

## 基本概念

### 狭义理解

在一个进程中，有且仅有一个线程在执行任务，每个任务都必须等待前一个任务完成后才能开始执行。

### 广义理解

在一个进程中，针对某一类型的任务，有且仅有一个线程在执行，每个任务都必须等待前一个任务完成后才能开始执行。也就是说，一个进程中可以同时有 N 个线程处理 N 类不同类型的任务，这些任务间彼此没有依赖关系和资源共享问题，此时某个线程可以看作是某类任务的单线程。

## 核心特点

- **简单易用：** 单线程模型相对简单，易于理解和实现
- **顺序执行：** 所有任务在同一线程中顺序执行，不存在并发问题
- **阻塞执行：** 如果一个任务阻塞了线程，整个进程就会被阻塞
- **资源共享：** 由于只有一个线程存在，不需要考虑资源共享和同步的问题

## 适用场景

- 严格要求执行顺序的场景
- 高并发低延迟的场景
- 非阻塞 IO 密集型场景
- 帧同步游戏
- 状态同步游戏

## 使用方式

### 基本模式

在单线程模型下，路由处理器函数直接在主线程中执行，无需额外操作：

```go
func greetHandler(ctx node.Context) {
    req := &greetReq{}
    res := &greetRes{}

    ctx.Defer(func() {
        if err := ctx.Response(res); err != nil {
            log.Errorf("response message failed: %v", err)
        }
    })

    if err := ctx.Parse(req); err != nil {
        log.Errorf("parse request message failed: %v", err)
        res.Code = codes.InternalError.Code()
        return
    }

    log.Info(req.Message)

    res.Code = codes.OK.Code()
    res.Message = fmt.Sprintf("current time: %s", xtime.Now().Format(xtime.DateTime))
}
```

### 关键点

1. **不使用 `ctx.Task()`**：所有逻辑直接在路由处理器中执行
2. **无并发问题**：不需要加锁或使用同步原语
3. **避免阻塞操作**：不要在处理器中执行阻塞操作（如数据库查询、远程调用），否则会阻塞整个线程

## 完整示例

### 项目初始化

```bash
mkdir single-thread-example
cd single-thread-example
go mod init single-thread-example
go get github.com/dobyte/due/v2@v2.4.2
go get github.com/dobyte/due/locate/redis/v2@e5cd009
go get github.com/dobyte/due/registry/consul/v2@e5cd009
```

### 主程序

```go
package main

import (
    "fmt"

    "github.com/dobyte/due/locate/redis/v2"
    "github.com/dobyte/due/registry/consul/v2"
    "github.com/dobyte/due/v2"
    "github.com/dobyte/due/v2/cluster/node"
    "github.com/dobyte/due/v2/codes"
    "github.com/dobyte/due/v2/log"
    "github.com/dobyte/due/v2/utils/xtime"
)

const greet = 1

func main() {
    container := due.NewContainer()
    locator := redis.NewLocator()
    registry := consul.NewRegistry()
    component := node.NewNode(
        node.WithLocator(locator),
        node.WithRegistry(registry),
    )
    initApp(component.Proxy())
    container.Add(component)
    container.Serve()
}

func initApp(proxy *node.Proxy) {
    proxy.Router().AddRouteHandler(greet, greetHandler)
}

type greetReq struct {
    Message string `json:"message"`
}

type greetRes struct {
    Code    int    `json:"code"`
    Message string `json:"message"`
}

// 所有路由处理器在同一线程中顺序执行，不存在并发问题
func greetHandler(ctx node.Context) {
    req := &greetReq{}
    res := &greetRes{}
    ctx.Defer(func() {
        if err := ctx.Response(res); err != nil {
            log.Errorf("response message failed: %v", err)
        }
    })

    if err := ctx.Parse(req); err != nil {
        log.Errorf("parse request message failed: %v", err)
        res.Code = codes.InternalError.Code()
        return
    }

    log.Info(req.Message)

    res.Code = codes.OK.Code()
    res.Message = fmt.Sprintf("I'm tcp server, and the current time is: %s", xtime.Now().Format(xtime.DateTime))
}
```

## 注意事项

1. **禁止阻塞操作**：在单线程模型中，任何阻塞操作都会导致整个进程阻塞，应避免数据库查询、远程 RPC 调用等阻塞操作
2. **无需同步**：不需要使用 mutex、channel 等同步原语保护共享数据
3. **顺序保证**：所有消息严格按照到达顺序处理
4. **性能瓶颈**：单线程模型的吞吐量受限于单个线程的处理能力

## 与其他模型的对比

| 特性 | 单线程模型 | 多协程模型 | Actor 模型 |
|------|-----------|-----------|-----------|
| 执行顺序 | 严格有序 | 无序 | Actor 内有序 |
| 并发安全 | 天然安全 | 需同步原语 | 天然安全 |
| 阻塞影响 | 阻塞整个进程 | 仅阻塞当前协程 | 仅阻塞当前 Actor |
| 适用场景 | 帧同步/状态同步 | IO 密集型 | 房间类游戏 |
| 实现复杂度 | 低 | 中 | 中高 |
