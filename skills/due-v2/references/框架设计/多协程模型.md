# 多协程模型

## 概述

Due v2 框架支持多协程并发模型，通过 `ctx.Task()` 将任务投递到协程池中处理。多协程模型适用于不考虑执行顺序、阻塞 IO 密集型等场景，如数据库操作和远程服务调用。

## 基本概念

多协程模型将原本在单线程中的任务通过 `ctx.Task()` 投递到协程池中进行处理，任务处理顺序由有序变为无序，并发处理需要使用同步原语。

## 核心特点

- **并发执行：** 任务在协程池中并发执行，提升吞吐量
- **无序处理：** 任务执行顺序不保证，后到的任务可能先完成
- **需要同步：** 共享数据访问需使用 mutex、channel 等同步原语
- **非阻塞主线程：** 阻塞操作不会影响主线程处理其他消息

## 适用场景

- 不考虑执行顺序的场景
- 阻塞 IO 密集型场景
- 数据库操作
- 远程服务调用（RPC、HTTP）
- 文件读写操作

## 使用方式

### 核心 API：ctx.Task()

通过 `ctx.Task()` 将业务逻辑投递到协程池执行：

```go
func greetHandler(ctx node.Context) {
    ctx.Task(func(ctx node.Context) {
        // 这里的代码在协程池中并发执行
        req := &greetReq{}
        res := &greetRes{}

        ctx.Defer(func() {
            if err := ctx.Response(res); err != nil {
                log.Errorf("response message failed: %v", err)
            }
        })

        if err := ctx.Parse(req); err != nil {
            log.Errorf("parse request message failed: %v", err)
            res.Code = codes.InternalError.Code()
            return
        }

        // 可以安全地执行阻塞操作
        // 如数据库查询、远程调用等
        result, err := db.Query(...)
        if err != nil {
            res.Code = codes.InternalError.Code()
            return
        }

        res.Code = codes.OK.Code()
        res.Message = result
    })
}
```

### 关键点

1. **使用 `ctx.Task()`**：将业务逻辑包装在 Task 回调中
2. **回调签名**：`func(ctx node.Context)`，回调中的 ctx 与原 ctx 一致
3. **并发安全**：访问共享变量时需使用同步原语
4. **ctx.Defer()**：在 Task 内部同样可以使用 Defer 确保响应发送


### 主程序

```go
package main

import (
    "fmt"

    "github.com/dobyte/due/locate/redis/v2"
    "github.com/dobyte/due/registry/consul/v2"
    "github.com/dobyte/due/v2"
    "github.com/dobyte/due/v2/cluster/node"
    "github.com/dobyte/due/v2/codes"
    "github.com/dobyte/due/v2/log"
    "github.com/dobyte/due/v2/utils/xtime"
)

const greet = 1

func main() {
    container := due.NewContainer()
    locator := redis.NewLocator()
    registry := consul.NewRegistry()
    component := node.NewNode(
        node.WithLocator(locator),
        node.WithRegistry(registry),
    )
    initApp(component.Proxy())
    container.Add(component)
    container.Serve()
}

func initApp(proxy *node.Proxy) {
    proxy.Router().AddRouteHandler(greet, greetHandler)
}

type greetReq struct {
    Message string `json:"message"`
}

type greetRes struct {
    Code    int    `json:"code"`
    Message string `json:"message"`
}

// 通过 ctx.Task 投递到协程池中处理，任务处理无序，并发处理需使用同步原语
func greetHandler(ctx node.Context) {
    ctx.Task(func(ctx node.Context) {
        req := &greetReq{}
        res := &greetRes{}
        ctx.Defer(func() {
            if err := ctx.Response(res); err != nil {
                log.Errorf("response message failed: %v", err)
            }
        })

        if err := ctx.Parse(req); err != nil {
            log.Errorf("parse request message failed: %v", err)
            res.Code = codes.InternalError.Code()
            return
        }

        log.Info(req.Message)

        res.Code = codes.OK.Code()
        res.Message = fmt.Sprintf("I'm tcp server, and the current time is: %s", xtime.Now().Format(xtime.DateTime))
    })
}
```

## 注意事项

1. **并发安全**：访问共享变量（如全局 map、计数器）时需加锁或使用 channel
2. **响应时机**：`ctx.Response()` 需在 Task 回调内部调用，不能在外部提前调用
3. **ctx.Defer()**：建议在 Task 回调内部使用 Defer 确保响应一定被发送
4. **错误处理**：Task 回调中的 panic 不会影响主线程，但需做好错误捕获

## 与其他模型的对比

| 特性 | 单线程模型 | 多协程模型 | Actor 模型 |
|------|-----------|-----------|-----------|
| 执行顺序 | 严格有序 | 无序 | Actor 内有序 |
| 并发安全 | 天然安全 | 需同步原语 | 天然安全 |
| 阻塞影响 | 阻塞整个进程 | 仅阻塞当前协程 | 仅阻塞当前 Actor |
| 适用场景 | 帧同步/状态同步 | IO 密集型 | 房间类游戏 |
| 实现复杂度 | 低 | 中 | 中高 |

## 混合使用

在实际项目中，可以在同一路由处理器中混合使用单线程和多协程模式：

- 需要顺序保证的逻辑（如参数校验、权限检查）直接在处理器中执行
- 阻塞型逻辑（如数据库查询）通过 `ctx.Task()` 投递到协程池
