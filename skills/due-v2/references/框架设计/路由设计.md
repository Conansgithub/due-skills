# 路由设计

## 概述

路由（Route）是消息在 Due 集群中流转的核心标识。每个消息都携带一个路由号，网关根据路由号将消息分发到对应的节点服进行处理。Due 支持两种路由模式：无状态路由和有状态路由。

## 路由注册

### 注册机制

路由处理器在节点服启动时注册，并通过注册中心同步到网关服：

```go
// 节点服启动时注册路由
func initApp(proxy *node.Proxy) {
    // 注册单个路由
    proxy.Router().AddRouteHandler(1, loginHandler)
    proxy.Router().AddRouteHandler(2, moveHandler)
    
    // 注册路由组
    proxy.Router().Group(func(group *node.RouterGroup) {
        group.AddRouteHandler(10, chatHandler)
        group.AddRouteHandler(11, friendHandler)
    })
}
```

### 路由信息同步

```
节点服启动
    ↓
注册路由处理器（route=1,2,10,11）
    ↓
将路由信息注册到注册中心
    ↓
网关服通过注册中心发现节点服
    ↓
网关服更新本地路由表
    ↓
网关服可以分发路由号 1,2,10,11 的消息到该节点服
```

## 无状态路由（Stateless Route）

### 概念

无状态路由类似于 HTTP 路由，消息可以被分发到任意一个注册了该路由的节点服。适用于：
- 登录验证
- 匹配系统
- 聊天服务
- 无状态查询

### 注册方式

```go
// 默认就是无状态路由
proxy.Router().AddRouteHandler(1, loginHandler)

// 显式指定无状态路由
proxy.Router().AddRouteHandler(1, loginHandler, node.StatelessRoute)
```

### 分发策略

网关服支持三种无状态路由分发策略：

#### 1. 随机分发（random）

```toml
[cluster.gate]
  dispatch = "random"  # 默认策略
```

**工作原理：**
- 网关从注册了该路由的节点服中随机选择一个
- 简单高效，负载相对均衡

**示例：**
```
客户端发送 route=1 的消息
    ↓
网关服发现节点 A、B、C 都注册了 route=1
    ↓
随机选择节点 B
    ↓
消息转发到节点 B 处理
```

#### 2. 轮询分发（round-robin）

```toml
[cluster.gate]
  dispatch = "round-robin"
```

**工作原理：**
- 按顺序轮流分发到各个节点服
- 保证每个节点服均匀接收消息

**示例：**
```
第 1 个 route=1 消息 → 节点 A
第 2 个 route=1 消息 → 节点 B
第 3 个 route=1 消息 → 节点 C
第 4 个 route=1 消息 → 节点 A
...
```

#### 3. 一致性哈希（consistent-hash）

```toml
[cluster.gate]
  dispatch = "consistent-hash"
```

**工作原理：**
- 根据消息的某个字段（如用户 ID）计算哈希值
- 相同哈希值的消息总是分发到同一个节点服
- 适用于需要保持会话亲和性的场景

**示例：**
```
用户 1001 的消息 → 哈希计算 → 节点 A
用户 1002 的消息 → 哈希计算 → 节点 B
用户 1001 的后续消息 → 哈希计算 → 仍然是节点 A
```

### 无状态路由示例

```go
// 登录处理器（无状态路由）
func loginHandler(ctx node.Context) {
    req := &loginReq{}
    if err := ctx.Parse(req); err != nil {
        return
    }
    
    // 验证账号密码
    uid, err := verifyUser(req.Account, req.Password)
    if err != nil {
        ctx.Response(&loginRes{Code: 401})
        return
    }
    
    // 绑定网关，建立用户连接
    if err := ctx.BindGate(uid); err != nil {
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    ctx.Response(&loginRes{Code: 0, UID: uid})
}
```

## 有状态路由（Stateful Route）

### 概念

有状态路由要求消息必须分发到用户绑定的节点服。适用于：
- 游戏逻辑处理
- 角色状态管理
- 战斗系统
- 需要访问用户内存数据的场景

### 注册方式

```go
// 显式指定有状态路由
proxy.Router().AddRouteHandler(2, moveHandler, node.StatefulRoute)
```

### 工作原理

#### 1. 用户绑定节点

用户登录后，需要绑定到一个节点服：

```go
// 登录成功后绑定节点
func loginHandler(ctx node.Context) {
    // ... 验证逻辑 ...
    
    // 绑定网关
    ctx.BindGate(uid)
    
    // 绑定节点（将用户绑定到当前节点服）
    ctx.BindNode(uid)
}
```

绑定信息存储在 Redis 中：
```
Key: due:locate:user:{uid}
Value: {nodeID: "node-1", gateID: "gate-1", connID: 12345}
```

#### 2. 消息路由

```
客户端发送 route=2 的消息（携带 UID）
    ↓
网关服检查路由类型：有状态路由
    ↓
网关服查询用户绑定信息：UID → Node-1
    ↓
网关服将消息转发到 Node-1
    ↓
Node-1 处理消息（可以访问用户内存数据）
```

### 有状态路由示例

```go
// 移动处理器（有状态路由）
func moveHandler(ctx node.Context) {
    uid := ctx.UID()  // 获取用户 ID
    
    // 从内存中获取玩家数据（只有绑定的节点服才能访问）
    player := getPlayerFromMemory(uid)
    if player == nil {
        ctx.Response(&moveRes{Code: 404})
        return
    }
    
    req := &moveReq{}
    if err := ctx.Parse(req); err != nil {
        return
    }
    
    // 更新玩家位置
    player.X = req.X
    player.Y = req.Y
    
    ctx.Response(&moveRes{Code: 0})
}
```

## 路由模式对比

| 特性 | 无状态路由 | 有状态路由 |
|------|-----------|-----------|
| **消息分发** | 任意节点服 | 用户绑定的节点服 |
| **数据访问** | 无状态数据（数据库/缓存） | 有状态数据（内存） |
| **负载均衡** | 自动均衡 | 依赖用户分布 |
| **适用场景** | 登录、匹配、聊天 | 游戏逻辑、战斗 |
| **性能** | 高（无需查询绑定） | 中（需查询绑定信息） |
| **扩展性** | 优秀 | 受限于用户绑定 |

## 混合使用

实际项目中，通常混合使用两种路由模式：

```go
func initApp(proxy *node.Proxy) {
    proxy.Router().Group(func(group *node.RouterGroup) {
        // 无状态路由：登录、匹配
        group.AddRouteHandler(1, loginHandler)        // 无状态
        group.AddRouteHandler(2, matchHandler)        // 无状态
        
        // 有状态路由：游戏逻辑
        group.AddRouteHandler(10, moveHandler, node.StatefulRoute)
        group.AddRouteHandler(11, attackHandler, node.StatefulRoute)
        group.AddRouteHandler(12, skillHandler, node.StatefulRoute)
    })
}
```

## 路由表管理

### 网关服路由表

网关服维护一张路由表，记录每个路由号对应的节点服列表：

```go
type RouteTable struct {
    // 无状态路由：路由号 → 节点服列表
    statelessRoutes map[uint32][]*NodeInfo
    
    // 有状态路由：路由号 → 节点服列表（用于验证）
    statefulRoutes map[uint32][]*NodeInfo
}

type NodeInfo struct {
    NodeID string
    Addr   string
    Weight int  // 权重（用于负载均衡）
}
```

### 路由表更新

```
节点服 A 启动，注册路由 [1, 2, 10]
    ↓
注册中心通知网关服
    ↓
网关服更新路由表：
  route 1 → [Node-A]
  route 2 → [Node-A]
  route 10 → [Node-A]

节点服 B 启动，注册路由 [1, 2, 10]
    ↓
网关服更新路由表：
  route 1 → [Node-A, Node-B]
  route 2 → [Node-A, Node-B]
  route 10 → [Node-A, Node-B]

节点服 A 下线
    ↓
网关服更新路由表：
  route 1 → [Node-B]
  route 2 → [Node-B]
  route 10 → [Node-B]
```

## 路由中间件

Due 支持路由中间件，可以在消息处理前执行通用逻辑：

```go
// 认证中间件
func authMiddleware(ctx node.Context) {
    uid := ctx.UID()
    if uid == 0 {
        ctx.Response(&baseRes{Code: 401})
        ctx.Abort()  // 中止后续处理
        return
    }
    ctx.Next()  // 继续处理
}

// 日志中间件
func logMiddleware(ctx node.Context) {
    log.Printf("收到消息: route=%d, uid=%d", ctx.Route(), ctx.UID())
    ctx.Next()
}

// 注册中间件
proxy.Router().Use(logMiddleware)  // 全局中间件

proxy.Router().Group(func(group *node.RouterGroup) {
    group.Use(authMiddleware)  // 路由组中间件
    group.AddRouteHandler(10, moveHandler, node.StatefulRoute)
})
```

## 最佳实践

### 1. 合理划分路由号

```go
// 按功能模块划分路由号范围
const (
    // 登录模块：1-99
    RouteLogin  = 1
    RouteLogout = 2
    
    // 匹配模块：100-199
    RouteMatchStart = 100
    RouteMatchCancel = 101
    
    // 游戏模块：1000-1999
    RouteMove   = 1000
    RouteAttack = 1001
    RouteSkill  = 1002
)
```

### 2. 路由号集中管理

```go
// routes/routes.go
package routes

const (
    Login  uint32 = 1
    Logout uint32 = 2
    Move   uint32 = 1000
    Attack uint32 = 1001
)

// 路由号到处理器名的映射（用于调试）
var RouteNames = map[uint32]string{
    Login:  "Login",
    Logout: "Logout",
    Move:   "Move",
    Attack: "Attack",

```

### 3. 路由性能优化

```go
// 避免在路由处理器中执行耗时操作
func badHandler(ctx node.Context) {
    // ❌ 错误：阻塞路由处理
    result := heavyComputation()  // 耗时操作
    ctx.Response(result)
}

func goodHandler(ctx node.Context) {
    // ✅ 正确：异步处理耗时操作
    go func() {
        result := heavyComputation()
        ctx.AsyncResponse(result)
    }()
}
```

## 常见问题

### Q1: 如何知道一个路由号注册在哪些节点服？

**A:** 可以通过注册中心查询，或者在网关服日志中查看路由表更新记录。

### Q2: 有状态路由的用户绑定信息丢失怎么办？

**A:** 绑定信息存储在 Redis 中，如果 Redis 数据丢失，用户需要重新登录并绑定节点。

### Q3: 能否动态添加或删除路由？

**A:** 可以，但需要重启节点服。建议在启动时注册所有路由。

### Q4: 路由号有数量限制吗？

**A:** 取决于 `routeBytes` 配置：
- 1 字节：最多 255 个路由
- 2 字节：最多 65535 个路由
- 4 字节：最多 40 亿个路由

## 总结

路由设计是 Due 分布式架构的核心，通过无状态路由和有状态路由的灵活组合，可以构建高性能、可扩展的游戏服务器。理解路由机制，是开发复杂游戏逻辑的基础。
