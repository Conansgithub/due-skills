# 绑定节点

## 概述

绑定节点（Bind Node）是将用户（UID）与特定节点服建立关联的过程。绑定后，该用户的所有有状态路由消息都会被路由到绑定的节点服处理。这是实现有状态服务的基础。

## 核心概念

### 为什么需要绑定节点

```
场景：MMORPG 游戏

玩家 A 登录到节点服 1
    ↓
玩家 A 的游戏数据（角色、背包、位置）加载到节点服 1 的内存中
    ↓
玩家 A 发送移动消息
    ↓
消息必须路由到节点服 1（因为只有节点服 1 有玩家 A 的内存数据）
    ↓
如果路由到节点服 2，节点服 2 没有玩家 A 的数据，无法处理
```

### 绑定信息

绑定信息存储在 Redis 中：

```go
type NodeBinding struct {
    UID    int64  `json:"uid"`    // 用户 ID
    NodeID string `json:"nodeID"` // 节点服 ID
}
```

Redis Key 格式：
```
due:locate:user:{uid}
```

## 绑定流程

### 完整流程

```
1. 用户登录，绑定网关（BindGate）
   ↓
2. 节点服加载用户数据到内存
   ↓
3. 节点服调用 ctx.BindNode(uid)
   ↓
4. 绑定信息写入 Redis：UID → NodeID
   ↓
5. 后续有状态路由消息根据绑定信息路由到该节点服
```

### 代码示例

```go
// 登录处理器
func loginHandler(ctx node.Context) {
    req := &loginReq{}
    if err := ctx.Parse(req); err != nil {
        ctx.Response(&loginRes{Code: 400})
        return
    }
    
    // 1. 验证账号密码
    uid, err := verifyUser(req.Account, req.Password)
    if err != nil {
        ctx.Response(&loginRes{Code: 401})
        return
    }
    
    // 2. 绑定网关
    if err := ctx.BindGate(uid); err != nil {
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    // 3. 绑定节点（关键步骤）
    if err := ctx.BindNode(uid); err != nil {
        ctx.UnbindGate(uid)
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    // 4. 加载玩家数据到内存
    player, err := loadPlayerData(uid)
    if err != nil {
        ctx.UnbindNode(uid)
        ctx.UnbindGate(uid)
        ctx.Response(&loginRes{Code: 500})
        return
    }
    
    // 5. 将玩家对象存入内存
    playerManager.AddPlayer(uid, player)
    
    ctx.Response(&loginRes{Code: 0, UID: uid})
}
```

## 绑定机制详解

### BindNode 方法

```go
func (ctx *Context) BindNode(uid int64) error
```

**参数：**
- `uid`：用户 ID

**返回值：**
- `error`：绑定失败时返回错误

**内部实现：**

```go
func (ctx *Context) BindNode(uid int64) error {
    // 1. 检查 UID 合法性
    if uid <= 0 {
        return errors.New("UID 必须大于 0")
    }
    
    // 2. 检查是否已绑定
    existing, err := ctx.locator.LocateNode(uid)
    if err == nil && existing != nil {
        if existing.NodeID == ctx.node.ID() {
            return nil  // 已绑定到当前节点，无需重复绑定
        }
        // 已绑定到其他节点，先解绑
        ctx.locator.UnbindNode(uid)
    }
    
    // 3. 写入绑定信息到 Redis
    binding := &NodeBinding{
        UID:    uid,
        NodeID: ctx.node.ID(),
    }
    if err := ctx.locator.BindNode(uid, binding); err != nil {
        return err
    }
    
    return nil
}
```

### 消息路由机制

#### 有状态路由

```
客户端发送消息（route=10, UID=1001）
    ↓
网关服检查路由类型：有状态路由
    ↓
网关服查询绑定信息：UID=1001 → Node-1
    ↓
网关服将消息转发到 Node-1
    ↓
Node-1 处理消息（可以访问内存中的玩家数据）
```

#### 无状态路由

```
客户端发送消息（route=1, UID=1001）
    ↓
网关服检查路由类型：无状态路由
    ↓
网关服忽略绑定信息，按分发策略选择节点服
    ↓
消息可能路由到 Node-1、Node-2 或 Node-3
    ↓
节点服处理消息（不能访问内存数据，需从数据库/缓存读取）
```

## 解绑节点

### 自动解绑

节点服关闭时，自动解绑所有用户：

```go
// 节点服关闭回调
func (n *Node) onShutdown() {
    // 获取所有绑定的用户
    uids := n.locator.ListBoundUsers(n.ID())
    
    // 批量解绑
    for _, uid := range uids {
        n.locator.UnbindNode(uid)
    }
}
```

### 手动解绑

用户登出时，主动解绑：

```go
// 登出处理器
func logoutHandler(ctx node.Context) {
    uid := ctx.UID()
    
    // 1. 保存玩家数据到数据库
    player := playerManager.GetPlayer(uid)
    if player != nil {
        savePlayerData(player)
    }
    
    // 2. 从内存中移除玩家对象
    playerManager.RemovePlayer(uid)
    
    // 3. 解绑节点
    ctx.UnbindNode(uid)
    
    // 4. 解绑网关
    ctx.UnbindGate(uid)
    
    ctx.Response(&logoutRes{Code: 0})
}
```

## 跨节点通信

### 场景：玩家 A 攻击玩家 B

```
玩家 A（绑定在 Node-1）攻击玩家 B（绑定在 Node-2）
    ↓
Node-1 处理攻击逻辑
    ↓
Node-1 需要通知 Node-2 扣除玩家 B 的血量
    ↓
Node-1 通过 RPC 调用 Node-2
    ↓
Node-2 扣除玩家 B 血量并广播伤害信息
```

### 代码示例

```go
// Node-1：攻击处理器
func attackHandler(ctx node.Context) {
    attackerUID := ctx.UID()
    req := &attackReq{}
    if err := ctx.Parse(req); err != nil {
        return
    }
    
    targetUID := req.TargetUID
    
    // 1. 查询目标玩家所在节点
    binding, err := ctx.LocateNode(targetUID)
    if err != nil {
        ctx.Response(&attackRes{Code: 404, Message: "目标玩家不存在"})
        return
    }
    
    // 2. 计算伤害
    attacker := playerManager.GetPlayer(attackerUID)
    damage := calculateDamage(attacker)
    
    // 3. 通过 RPC 调用目标节点服
    if binding.NodeID != ctx.node.ID() {
        // 跨节点调用
        err := ctx.node.Call(binding.NodeID, "applyDamage", &DamageReq{
            TargetUID: targetUID,
            Damage:    damage,
        })
        if err != nil {
            ctx.Response(&attackRes{Code: 500, Message: "攻击失败"})
            return
        }
    } else {
        // 同节点调用
        target := playerManager.GetPlayer(targetUID)
        target.HP -= damage
    }
    
    ctx.Response(&attackRes{Code: 0, Damage: damage})
}

// Node-2：应用伤害 RPC 方法
func (n *Node) applyDamage(req *DamageReq) error {
    target := playerManager.GetPlayer(req.TargetUID)
    if target == nil {
        return errors.New("玩家不存在")
    }
    
    target.HP -= req.Damage
    
    // 广播伤害信息
    broadcastDamage(target.UID, req.Damage)
    
    return nil
}
```

## 负载均衡与绑定

### 问题：节点服负载不均

```
场景：1000 个玩家登录

Node-1：绑定 800 个玩家（负载过高）
Node-2：绑定 150 个玩家
Node-3：绑定 50 个玩家
```

### 解决方案：智能绑定

```go
// 登录时选择负载最低的节点服
func loginHandler(ctx node.Context) {
    uid := generateUID()
    
    // 绑定网关
    ctx.BindGate(uid)
    
    // 获取所有节点服的负载信息
    nodes := ctx.registry.ListNodes()
    
    // 选择负载最低的节点服
    targetNode := selectLeastLoadedNode(nodes)
    
    // 如果目标节点服不是当前节点服，转发绑定请求
    if targetNode.ID != ctx.node.ID() {
        ctx.node.Call(targetNode.ID, "bindUser", &BindReq{UID: uid})
    } else {
        ctx.BindNode(uid)
        loadPlayerData(uid)
    }
}
```

## 最佳实践

### 1. 绑定前检查

```go
func bindUser(ctx node.Context, uid int64) error {
    // 检查用户是否已绑定到其他节点
    existing, _ := ctx.LocateNode(uid)
    if existing != nil && existing.NodeID != ctx.node.ID() {
        log.Warnf("用户 %d 已绑定到节点 %s，先解绑", uid, existing.NodeID)
        ctx.locator.UnbindNode(uid)
    }
    
    return ctx.BindNode(uid)
}
```

### 2. 绑定后加载数据

```go
func loginHandler(ctx node.Context) {
    uid := ctx.UID()
    
    // 绑定节点
    ctx.BindNode(uid)
    
    // 立即加载数据到内存
    player, err := loadPlayerFromDB(uid)
    if err != nil {
        ctx.UnbindNode(uid)
        return err
    }
    
    playerManager.AddPlayer(uid, player)
}
```

### 3. 定期保存数据

```go
// 每 5 分钟保存一次所有玩家数据
func startAutoSave() {
    ticker := time.NewTicker(5 * time.Minute)
    go func() {
        for range ticker.C {
            for _, player := range playerManager.GetAllPlayers() {
                savePlayerToDB(player)
            }
        }
    }()
}
```

### 4. 节点服优雅关闭

```go
func (n *Node) onShutdown() {
    log.Info("节点服准备关闭，保存所有玩家数据...")
    
    // 1. 停止接受新连接
    n.stopAccepting()
    
    // 2. 保存所有玩家数据
    for _, player := range playerManager.GetAllPlayers() {
        savePlayerToDB(player)
    }
    
    // 3. 解绑所有用户
    for uid := range playerManager.GetAllPlayers() {
        n.locator.UnbindNode(uid)
    }
    
    log.Info("节点服关闭完成")
}
```

## 常见问题

### Q1: 绑定节点后可以更换节点吗？

**A:** 可以，但需要：
1. 保存当前节点的数据
2. 解绑当前节点
3. 绑定到新节点
4. 在新节点加载数据

这通常用于节点服负载均衡或故障转移。

### Q2: 节点服宕机，绑定的用户怎么办？

**A:** 
- 绑定信息仍存在于 Redis 中
- 用户需要重新登录，绑定到新的节点服
- 建议实现自动重连机制

### Q3: 如何实现跨服战斗？

**A:** 跨服战斗需要：
1. 所有参战玩家绑定到同一个节点服（战斗服）
2. 或者通过 RPC 实现跨节点通信
3. 使用 Actor 模型管理战斗房间

### Q4: 绑定信息会过期吗？

**A:** 默认不会过期，但建议设置 TTL：
```go
// Redis 绑定信息设置过期时间（如 24 小时）
binding := &NodeBinding{UID: uid, NodeID: nodeID}
redis.Set(key, binding, 24*time.Hour)
```

## 总结

绑定节点是实现有状态服务的核心机制，通过将用户与节点服关联，确保消息路由的正确性。合理使用绑定机制，可以构建高性能、可扩展的游戏服务器。
